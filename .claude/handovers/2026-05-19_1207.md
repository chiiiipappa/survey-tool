## 決定事項

- **差し替えボタンは `<label for="...">` で実装**: JS の `replaceInput.click()` は `pointer-events:none` 付き hidden input に対し一部ブラウザで動作しない。STEP2 の同一実装では問題なかったことから環境依存の挙動と判断。`<label>` は JS 不要でブラウザネイティブ動作のため確実
- **JS キャッシュ無効化は開発用途に限定**: `_NoCacheStaticFiles` は開発ツールの性質上、パフォーマンスより動作確認の確実性を優先
- **テスト用サンプルデータを追加**: `sample_data/layout_sample_fanlabel.csv`（＊ファンラベル設問入り）をリポジトリに追加

---

## 捨てた選択肢と理由

- **`?v=N` を全インポート URL に付与する案**: ES モジュールは URL 単位でキャッシュされるため、インポート元・先で URL が異なると別インスタンスになりシングルトン（AppState）が壊れる可能性がある。サーバー側ヘッダーで解決する方が安全
- **`pointer-events:none` を維持したまま `.click()` を呼ぶ案**: STEP2 と同一コードで STEP1 のみ動作しなかった。環境依存で再現性が低く、`<label>` 方式が確実なため採用しなかった
- **`e.stopPropagation()` の追加**: ボタンが drop zone の外にいるため不要。追加しても問題はないが、混乱を招くため削除

---

## ハマりどころ

- **STEP1 と STEP2 の差し替えボタンが同一コードなのに挙動が違う**: HTML 構造・CSS・JS コードがほぼ同一なのに STEP1 のみ動作しなかった。ブラウザの `pointer-events:none` input に対する programmatic click の扱いが環境依存であることが原因と推測
- **ES モジュールキャッシュ**: `index.html` に `?v=3` を付けても、インポートされる子モジュールは URL が変わらないためキャッシュが使われ続ける。修正が反映されないバグと混同しやすい
- **`btn-csv-replace` が drop zone の外にある**: 当初「ボタンが drop zone 内にあるため二重 click が原因」と誤診断した。HTML のインデントを精査して外にあることを確認した

---

## 学び

- `<label for="input-id">` はファイルピッカー起動の最も信頼性が高い方法。JS `.click()` は `opacity:0` / `pointer-events:none` の組み合わせでブラウザ・OS 依存の問題が出やすい
- FastAPI/Starlette の `StaticFiles` は `get_response` をオーバーライドすることでレスポンスヘッダーを追加できる
- ES モジュールのキャッシュバスティングは全インポートチェーンに波及するため、サーバー側の `Cache-Control` ヘッダーで対応するのが現実的
