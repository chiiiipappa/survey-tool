## 決定事項

- **親セットの `questionCodes` は常に空配列**。設問は子セット（カテゴリ）のみが持つ
- **カスタムセットは必ず `isParent: true`** の親セット構造で管理。フラット型は廃止
- **旧プロジェクトの後方互換**: `setLoadedProject` でマイグレーション（旧フラット型 → 「（未分類）」カテゴリ付き親）
- **setId 命名規則**:
  - 親: `parent_{timestamp}` 
  - 子: `child_{timestamp}`
  - 旧フラット型からのマイグレーション子: `{旧setId}_ch0`
  - 旧フラット型は `custom_` プレフィックス（後方互換で `isCustom` 判定に使用）
- **システムセット (`isCustom: false`) のデータ構造・表示は変更なし**
- **`<details>` の summary クリックはクロス集計を発火させない**（`e.target.closest("summary")` チェック）
- **バックエンド変更不要**: `question_sets` が `List[dict]` のため、新構造はそのまま保存・復元される

## 捨てた選択肢と理由

- **親セットをクリックしたらすべての子セットを一括集計**: 子セット単位で独立して集計する設計を優先。大量設問の一括集計が元々の問題だったため
- **作成UIを別画面に分離**: モーダル内のタブ構造を維持。既存の設問選択 UI（左右パネル）がそのまま使えるため

## ハマりどころ

- `_renderSidebar()` 内で `itemHtml` という関数名を `systemItemHtml` に改名した後、下部の `system.map(itemHtml)` が旧名のままで参照エラーになる箇所があった → 修正済み
- `index.html` の `step1-new-parent-form` に `style="display:none"` を2回書いてしまっていた → 修正済み

## 学び

- 旧データ構造から新構造へのマイグレーションは `setLoadedProject` に集約するのが最もシンプル。フロント側の他の場所で旧構造を意識しなくて済む
- `<details>/<summary>` で折りたたみ UI を実装する場合、`details[open]` 属性と `::before` CSS で矢印アニメーションが簡単に実現できる
- `node --check` でブラウザ非依存の JS 構文チェックが可能（import 文や DOM API は問題だが、構文エラーの早期検出に有効）
