## 決定事項

- 結果エリアはモード変化時のみクリア（`_prevStep3Mode` との比較で判定）。同じモード内での再描画（軸変更など）はクリアしない
- 通常分析キャッシュ（`_normalModeResultCache`）: `{ data, cacheKey }` 形式。`_crosstabCache` とは別に per-mode で管理
- 特定分析キャッシュ（`_specialModeResultCache`）: `{ data, modeTag }` 形式。生の API レスポンスを保持し、`_displaySpecialBlocks` を再呼び出しで復元
- 特定分析の生成結果（グラフ・集計表）はプロジェクトファイルに保存しない（大容量のため）。セッション内のみ保持
- 属性分析設定（`step3AttrSimpleCodes`, `step3AttrCrossPairs`）はプロジェクトファイルへの保存・復元を追加

## 捨てた選択肢と理由

- **DOM の innerHTML スナップショットで復元** → Chart.js の canvas が innerHTML に残らないため不採用
- **結果エリアを通常/特定で完全に分ける（別 div）** → HTML 構造の大改修が必要で既存コードへの影響大。`_clearResultArea()` によるクリアのほうが局所的
- **常に（statechange のたびに）結果クリア** → 軸変更や設問チェック変更でも結果が消えてしまうため不採用。モード変化のみをフックした

## ハマりどころ

- `step3-special-blocks-nav` の `_navInitialized` フラグが残ったままだと、復元時にナビのクリックイベントが再登録されないため、`_clearResultArea()` で `el._navInitialized = false` するのを忘れないこと

## 学び

- `_onStateChange` は statechange イベントのたびに呼ばれるため、「モードが変わった」かどうかを自前で追跡しないと意図しないタイミングでクリアが走る
- `_runSpecialAnalysis` で `await _displaySpecialBlocks(data, modeTag)` の後に `AppState.step3Mode` を読むと、非同期完了後のモードが取れる（通常は変わっていないが、ユーザーが高速操作した場合に備えてキャプチャするなら直前が安全）
