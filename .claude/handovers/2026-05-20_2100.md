## 決定事項

- **Parquet 保存先**: `tempfile.gettempdir() / "survey_parquet"` — プロジェクト外の OS 一時領域
- **`raw_data` dict は廃止**: アップロード後にどのエンドポイントからも読まれていないため、メモリ節約のため除去
- **`labeled_data` dict も廃止**: step3.py 以外の FA 系エンドポイントも全て parquet 経由に統一
- **列プロジェクション**: `target_question_codes` が指定されている場合のみ必要列のみ読む（空の場合は全列）
- **ファイルサイズ上限**: 50MB → 500MB（200MB ファイルの実用上の余裕を確保）
- **クリーンアップ**: 起動時に TTL（3600s）の2倍（7200s）より古いセッションディレクトリを削除

---

## 捨てた選択肢と理由

| 選択肢 | 理由 |
|--------|------|
| polars でクロス集計を書き直す | `_crosstab_sa` / `_crosstab_ma` を全面書き換えになり範囲が広い。pyarrow + pandas の列プロジェクションで十分な改善が見込める |
| `labeled_data` dict をキャッシュに残しつつ parquet も保存 | メモリ節約の目的と矛盾する。全て parquet 経由に統一する方がシンプル |
| TTLCache の expiry callback で parquet を削除 | `cachetools.TTLCache` は expiry callback 未対応。起動時クリーンアップで実用上十分 |
| キャッシュディレクトリをプロジェクト root の `cache/` に置く | OS の tmp ディレクトリの方が OS 管理・再起動クリーンアップが自動で行われる |

## 学び

- `pd.read_parquet(engine="pyarrow", columns=[...])` の列プロジェクションは、Parquet のカラムインデックスを活用するため大規模データで特に効果が高い
- MA 設問の列名は `question_code`（例: `Q3`）ではなく `bracket_columns[].display_header`（例: `Q3：選択肢名`）なので、`_resolve_needed_columns` で `bracket_map` を使ってマッピングが必要
- `df_to_serializable` は `step2.py` のアップロードフローでのみ使われていた（他ファイルには影響なし）
