## 捨てた選択肢と理由

| 選択肢 | 理由 |
|--------|------|
| **単一 JSON** | 大容量データでファイルが肥大化。将来 Parquet 追加時に形式変更が必要になる |
| **SQLite** | 依存追加は不要だが、ブラウザダウンロード経由の単ファイル運用には相性が悪い |
| **DuckDB** | 依存が重い（20MB+）、分析ツール本体に組み込むには大げさ |
| **labeled_data のみ保存** | `raw_data` は `codebook` から再生成できる理論だが、一部の数値型や空欄処理で不整合が生じうる。ユーザーが両方保存を選択 |

## ハマりどころ

- **日本語プロジェクト名を `Content-Disposition` ヘッダーに入れると `latin-1` エラー**
  - HTTP ヘッダーは latin-1 制限のため、ASCII-safe なフォールバック名と RFC 5987 の `filename*=UTF-8''...` を両方設定して解決
- **`Step2SaveData` の `default_factory` フィールドと Pydantic v2 の `model_fields` の扱い**
  - `model_fields[k].default` が `PydanticUndefined` になるケースがあり、型ヒントから list/dict/int/str を推定するフォールバック処理が必要だった
- **Python の `.isalnum()` は日本語を「英数字」と判定する**
  - `safe_name` 生成時に日本語がそのままファイル名に入りヘッダーエラー → `.isascii()` を追加して解決

## 学び

- HTTP `Content-Disposition` の日本語ファイル名は `filename*=UTF-8''...` + ASCII フォールバックの2段構えが正しい
- Python `str.isalnum()` は Unicode 文字（漢字等）も True を返す。ASCII 限定にするには `.isascii() and .isalnum()` を組み合わせる
- Pydantic v2 の `model_fields` を使った動的フィールド構築は `default_factory` の扱いに注意が必要
