### 自動カラー設定機能の実装

- 設問の選択肢数に応じてデフォルトの `valueColorMapping` を自動生成する機能を追加
- 対象: 選択肢数が **7・8・11 個**の設問
  - 7個 → `scale_67` 相当の 7色パレット（既存パレットと同色）
  - 8個（段階評価） → 新パレット 2-A（スケールキーワード検出で判定）
  - 8個（カテゴリ分類） → 新パレット 2-B
  - 11個 → `scale_1011` 相当の 11色パレット（既存パレットと同色）
- ファンラベル系（コアファン・ライトファン）は FIXED_PALETTES で対応のためスキップ

## 決定事項

- **保存先**: 自動カラーは `AppState.step3QuestionSettings[code].valueColorMapping` に保存（グローバル設定）
  - ビュー固有設定 `step3Views[viewId].questionSettings[code]` が存在する場合はそちらが優先される（既存のカラー解決ロジック通り）
- **上書きしない条件**: `step3QuestionSettings[code]?.valueColorMapping?.length` が truthy ならスキップ
- **再適用の挙動**: 「デフォルトカラーを再適用」はモーダル内の作業コピーを更新するだけ。実際の保存は「現在のグラフだけ変更」または「同じ集計軸すべてに適用」で行う
- **リセットの挙動**: 「この設問のカラーをリセット」は `clearQuestionColorState` を呼ぶ。アクティブなビューがある場合はビュー設定を削除し、グローバルの自動カラーにフォールバックする（完全クリアではない）

## 捨てた選択肢と理由

- **`applyAutoColorsIfUnset` を state.js 外（upload.js など）から呼ぶ**
  - 却下: 循環依存が生まれる可能性。state.js 内の `setUploadResult` / `setLoadedProject` で直接呼ぶほうがシンプル
- **自動カラーを `step3Views[viewId].questionSettings` に保存する**
  - 却下: ビューは軸ごとに存在し、自動カラーはどのビューでも共通に使うべき。グローバル設定に置くのが正しい
- **`computeAutoColorMapping` を step3.js 内だけに定義する**
  - 却下: step3.js から state.js を import している方向のため、逆方向依存を避けるため state.js に置いた
- **8選択肢の段階評価判定に機械学習/高度なヒューリスティック**
  - 却下: 要件のキーワード一覧（とても/やや/どちら等）で十分

## 学び

- `step3QuestionSettings[code].valueColorMapping` と `step3Views[viewId].questionSettings[code].valueColorMapping` の優先順位: ビュー側が勝つ。グローバルは fallback として機能する
- `_applyAutoColorsIfUnset` は `_emit()` の直前に呼べば emit 1回で済む（余分なイベント発火なし）
- pptx_builder.py の `_resolve_colors` は `valueColorMapping` を dict として扱っているが、実際は array of objects。これは既存バグで今回とは別問題
